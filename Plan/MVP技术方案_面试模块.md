# 项目学习-面试-履历 Agent —— MVP 技术方案（面试模块）

> 版本：MVP v1
> 日期：2026-07-11
> 范围：只做面试模块，命令行交互，单一测试项目（前端仓 + 后端仓）
> 技术栈：Mastra（编排）+ Vercel AI SDK（流式/模型接入）+ Qdrant（保留，MVP 可选）

---

## 一、MVP 定位与边界

这个 MVP 从完整版产品设计里抽取，只保留"面试 + 压力测试"这一条最小可跑闭环。目标有二：一是做成一个自己能用的面试陪练工具（用一个自己烂熟于心的项目做首个测试样本），二是作为投 Agent 工程师岗的作品，证明多 Agent 编排 + 工程化控制 + 效果评估的能力。

### 做什么

- 两个 Agent 角色：面试官 + 裁判
- 命令行一问一答交互
- 单一难度档（中级 / P6 感觉）
- 裁判用轻档案 + 按文件下钻（无护栏）
- 裁判结构化输出（Zod schema 约束）
- 面试官出题优先级规则：裁判有递点则优先问，无则自主出题
- 双评分：面试官主观分 + 裁判抗压分
- 面试存档：每场面试完整落盘（第几次、每轮问答、裁判判断）
- 一套亲手标注的回归测试集 + 评估脚本

### 砍什么（都是完整版的事）

学习模块（老师角色）、多领域适配、三档难度、三池配比（弱项/难点/随机）、下钻预算与标准护栏、自主下钻决策、跨场用户画像层、单场轻量断点、Web UI、自动生成简历候选、跨仓库链路自动识别。

---

## 二、核心设计原则（贯穿实现）

控制角色不靠 Prompt 自觉，靠四层硬约束（2026 生产级多 agent 共识）：

1. **信息边界靠工具注册**：面试官不注册任何读代码/读档案的工具，物理上够不到源码；裁判注册 `read_file` 工具。
2. **调用流程靠编排图**：角色之间不互相"叫人"，流转由编排层写死。
3. **输出结构靠结构化 schema**：裁判输出用 Zod schema 强制约束，防止格式漂移。
4. **Prompt 只管性格与判断力**：结构、边界、流程交给代码，Prompt 只负责面试官语气、裁判找接缝的判断标准。

一句话：每个角色被物理约束在自己的边界内，而不是靠它"记得"该干什么。

## 三、输入与数据准备

### 3.1 用户提供的输入

| 输入 | 形式 | 说明 |
|---|---|---|
| 前端仓路径 | 本地路径 | 被压测项目的前端仓库 |
| 后端仓路径 | 本地路径 | 被压测项目的后端仓库 |
| 简历一句话 | 手输字符串 | 面试官唯一起点，即简历上对该项目的一句话描述 |
| 目标岗位 JD | 手输文本 | 校准出题方向，面试官自主出题时使用 |
| 项目领域 | MVP 写死 = Agent/AI | 完整版才做领域选择 |
| 难度档 | MVP 写死 = 中级 | 完整版才做三档 |
| 前后端功能链路 | 手写配置文件 | 手动指认"前端某文件/函数 ↔ 后端某接口" |

### 3.2 预处理产出：轻档案（一次性生成）

独立预处理脚本扫两个仓库，生成一份结构化 JSON，小到能整个放进裁判上下文。

轻档案结构（示意）：

```json
{
  "project_name": "示例项目",
  "repos": [
    {
      "name": "frontend",
      "path": "/abs/path/frontend",
      "tree": "目录树（精简，忽略 node_modules 等）",
      "modules": [
        { "path": "src/stream/parser.ts", "role": "SSE 增量解析" }
      ]
    },
    {
      "name": "backend",
      "path": "/abs/path/backend",
      "tree": "目录树（精简）",
      "modules": [
        { "path": "api/stream_handler.py", "role": "SSE 接口，分块输出" }
      ]
    }
  ],
  "links": [
    {
      "desc": "流式输出链路",
      "frontend": "src/stream/parser.ts",
      "backend": "api/stream_handler.py"
    }
  ]
}
```

`modules` 的一句话职责可由预处理阶段用 LLM 扫描生成；`links` 由用户手写。

## 四、两个角色定义

### 4.1 面试官

- **输入**：简历句 + JD + 领域模板 + 对话历史 + 裁判本轮追问点（可能为空）
- **工具**：无（物理隔离于源码与档案）
- **输出**：一个面向用户的自然问题
- **出题规则（优先级）**：裁判本轮有递来追问点 → 优先问；无（第一轮，或裁判未下钻/未产出点）→ 自主出题，来源为 JD + 领域模板
- **Prompt 职责**：只写性格——一个中级 Agent 岗的资深行家面试官，追问自然、像真人，不上帝视角

### 4.2 裁判

- **输入**：轻档案 + 对话历史 + 用户最新回答
- **工具**：`read_file(路径)` —— 按需读取单个文件（MVP 无护栏、无预算，想读就读）
- **KPI**：压力测试用户说法的稳固度（不是抓造假）。真实代码是"出题弹药"，用于找出用户说法里最脆弱、最该被追问的接缝
- **输出**：强制 Zod schema 的结构化结果

裁判输出 schema（示意）：

```typescript
const JudgeOutput = z.object({
  robustness: z.enum(["solid", "partial", "collapsed"]), // 本轮回答的抗压判断
  collapse_point: z.string().nullable(),  // 崩在哪；solid 时为 null
  did_drill: z.boolean(),                  // 本轮是否下钻读了文件
  drilled_files: z.array(z.string()),      // 下钻读了哪些文件
  next_probe: z.string().nullable(),       // 递给面试官的下一个追问点；无则 null
  reasoning: z.string()                    // 判断理由（供复盘，不给面试官）
});
```

## 五、调度与状态

### 5.1 共享 State

一个对象贯穿整场面试：

```typescript
interface InterviewState {
  resume_line: string;        // 简历一句话
  jd: string;                 // 目标岗位 JD
  light_profile: object;      // 轻档案
  history: Turn[];            // 对话历史（面试官问 / 用户答 / 裁判判断）
  round: number;              // 当前轮次
  max_rounds: number;         // MVP 固定，如 8
}
```

### 5.2 固定流转（编排层写死，角色无权更改）

```
[第一轮] 裁判无点可递
  → 面试官自主出题（简历句 + JD + 领域模板）→ 抛开场问题
       ↓
     用户回答
       ↓
   裁判接管：读回答 →（必要时 read_file 下钻）→ 产出 JudgeOutput
       ↓
   编排层：round++；round <= max_rounds ?
       ├─ 是 → 面试官看 next_probe：有则优先问，无则自主出题 → 循环
       └─ 否 → 进入复盘
```

### 5.3 复盘输出

- 面试官主观分：讲得好不好（表达、逻辑、说服力）
- 裁判抗压分：各轮 robustness 汇总——扛到第几层、崩在哪、哪些全程稳住
- 两分 gap 诊断：主观高+抗压低=会包装一挖就崩（最危险）；主观低+抗压高=底子好但没讲出彩

## 六、面试存档

每场面试完整落盘为一个本地 JSON 文件，零数据库依赖。它是三样东西的原料：单场复盘、跨场进步追踪、以及完整版用户画像层的输入。MVP 只存不分析，但存的时候粒度留足，将来做画像分析不用回头补数据。

### 6.1 边界（避免和其他模块混淆）

- **不是用户画像层**：画像层是提炼后的规律（老崩在哪、哪些已过关），需跨场分析。存档只存原始记录，不做提炼——原始记录是画像层的原料。
- **不是轻量断点**：断点存的是"未完成的现场"（为中断续面）。存档存的是"已完成的存档"（整场结果）。时机与用途不同。
- 存档是独立的第三种东西，定位干净，不与上述两者冲突。

### 6.2 存档结构（每场一个 JSON）

```json
{
  "session_id": 3,                    // 第几次面试，自增序号
  "timestamp": "2026-07-11T15:30:00",
  "resume_line": "本场用的简历那句话",
  "jd": "本场用的目标岗位 JD",
  "project_name": "被测项目名",
  "max_rounds": 8,
  "turns": [
    {
      "round": 1,
      "question": "面试官问的问题",
      "question_source": "self",       // self=自主出题 | probe=裁判递点
      "answer": "用户的回答",
      "judge": {                        // 完整 JudgeOutput
        "robustness": "partial",
        "collapse_point": "……",
        "did_drill": true,
        "drilled_files": ["api/stream_handler.py"],
        "next_probe": "……",
        "reasoning": "……"
      }
    }
  ],
  "summary": {
    "subjective_score": "面试官主观分",
    "robustness_summary": "抗压分汇总",
    "gap_diagnosis": "两分 gap 诊断"
  }
}
```

### 6.3 存放约定

- 统一放在 `sessions/` 目录
- 文件名带序号 + 时间，如 `session_003_20260711.json`
- `session_id` 自增靠扫目录内已有文件的最大序号 +1，不需数据库维护计数器

## 七、评估方案（脱离主观判断的最小一层）

MVP 只做第一层：**标注测试集 + 回归脚本**。第二层（LLM-as-a-Judge）、第三层（分角色指标）留到调优阶段。

### 做法

1. 用熟悉的测试项目，手写 10-15 个测试案例，每个是 `{一段回答, 标注的正确判断}`，覆盖三类：真懂的扎实回答（应判 solid）、一挖就崩的虚回答（应判 collapsed）、能自圆其说的有效包装（应判 solid 或 partial）
2. 脚本把这些回答喂给裁判，对比它的 `robustness` 判断与标注，输出一致率
3. 每次改动裁判 Prompt / 换模型，跑一遍看一致率是否回归

### 为什么这是天然优势

你对测试项目烂熟于心，天然知道哪些说法真、哪些虚、哪些是有效包装——你自己就是最准的标注者。

### 投岗弹药（后续）

分角色指标（裁判抗压判断准确率 X%、面试官追问相关性、下钻精准度）是面试讲这个项目时的核心弹药——证明你不只会搭多 Agent 系统，还会衡量和保证质量。这是初级与中高级 Agent 工程师的分水岭。

## 八、技术栈与交付形态

- **编排**：Mastra（角色、工具、workflow 均为纯 TypeScript + Zod，serverless 友好，2026 TS agent 事实标准）
- **模型接入/流式**：Vercel AI SDK（Mastra 底层即走它，40+ provider）
- **结构化输出**：Mastra + Zod schema（对应"结构化解码"控制层）
- **向量库**：Qdrant 保留，MVP 阶段轻档案不大可不启用，视需要再接
- **交付形态**：纯后端 + 命令行。理由——自己用命令行足够；避开纯前端工作量，让展示焦点全部落在 Agent 编排能力上，契合"摆脱前端、证明后端/Agent 能力"的战略目标

命令行使用流程：跑预处理生成轻档案 → 启动面试 → 终端一问一答 → 结束输出双评分与复盘。

## 九、开工顺序建议

1. 预处理脚本：扫两个仓库生成轻档案 JSON（含手写 links）
2. 裁判角色：定 schema、接 `read_file` 工具、写判断 Prompt
3. 用标注测试集单独验证裁判判断质量（先别接面试官）
4. 面试官角色：写出题 Prompt、实现优先级规则
5. 编排层：串起固定流转 + 共享 State + 终止条件
6. 复盘输出：双评分汇总
7. 面试存档：整场结束写 `sessions/session_NNN_日期.json`
8. 回归脚本纳入日常，每次改动跑一遍

先让裁判单独可靠（第 2-3 步），再接面试官串流程——裁判是这个系统价值的核心，先把它验对了，后面才稳。
