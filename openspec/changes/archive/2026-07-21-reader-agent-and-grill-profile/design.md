## Context

前置变更 `preprocess-light-profile` 已实现并同步为主 spec(`openspec/specs/light-profile/spec.md`),其扫描、忽略清单、路径校验与 Zod 骨架可复用。本变更把这条脚本管道升格为一个**角色**,并把三角色的信息边界钉死。

约束:

- 技术栈已定:TypeScript + Zod;模型走 Vercel AI SDK + DeepSeek(`deepseek-chat`,`DEEPSEEK_API_KEY`)。
- `deepseek-chat` 上下文 **100K token**。WhatIf 双仓去噪后源码约 150K token,**装不下**——这是分批与在线问答两个设计的共同前提。
- 被测项目源码本地可读;简历与 JD 由用户以文件提供。
- 阅读者的产出是裁判的唯一输入,其形态一旦定下,`judge-agent-and-eval` 须同步修订。

实测基线(WhatIf 双仓,已收紧忽略清单后):

| | 文件数 | 体积 | ≈ token |
|---|---|---|---|
| frontend | 39 | 195 KB | 50K |
| backend | 105 | 392 KB | 100K |
| 目录树(两仓合计) | — | 3.4 KB | 850 |

## Goals / Non-Goals

**Goals:**

- 阅读者作为全知角色,离线产出一份**够指路**的项目地图,而非项目百科。
- 裁判在地图答不了时能问到全知者,且**问不到源码**——边界由返回值类型保证。
- 地图的盲区(未分析的批次)对下游可见,不被误读为"那里不存在东西"。
- 分批笔记可缓存,调汇总阶段的 Prompt 不必重付精读成本。

**Non-Goals:**

- 不做终评(需要面试官评价存在,属后续变更)。
- 不做面试官、编排层、复盘。
- 不引入向量库或代码索引——阅读者用 `glob`/`grep` 现场检索,不预建索引。
- 不做增量扫描(源码变更后重新全量生成即可);`notes/` 缓存只服务于同一次扫描内的重跑。
- 不做多轮对话式的阅读者——`ask_reader` 是无状态的一问一答。

## Decisions

**1. 「全知」归阅读者,不归裁判**

- 裁判只拿 `GRILL.md` + `profile.json`,不注册任何文件工具。
- 备选:给裁判 `glob`/`grep` 让它自行验证 —— 弃用。裁判的目标是**面试效果大于真实性**,它需要的是节奏与接缝,不是事实核查;而一旦它能读源码,"面试官隔离于源码"这条链上的信息落差就只剩投影函数一道闸。事实核查归属阅读者的终评环节(后续变更)。

**2. `ask_reader` 的返回值类型即边界**

- 签名 `ask_reader(question: string): Promise<string>`,返回自然语言结论。
- **不提供**返回源码、文件路径列表或结构化引用的模式。裁判想要代码也拿不到,因为接口里没有这条路径。
- 备选:允许阅读者按需附源码片段 —— 弃用。那等于裁判事实上全知,本变更的角色分层随之瓦解;而"允许但靠 Prompt 克制"正是方案第二节反对的"靠自觉"。
- 剩余风险:阅读者可能把源码逐字**转述**进自然语言答案。这一条与 `next_probe` 的文本泄露同类,代码无法可靠强制,靠 Prompt 约束 + 评估期人工抽查。

**3. 产出是目录而非单个 JSON,且 `GRILL.md` 必须是真文件**

```
<输出目录>/
  GRILL.md        地图正文 + 目录树附录   → 裁判整篇读
  profile.json    metadata                → 编排层与后续角色按字段取用
  notes/*.json    分批笔记                 → 不进任何 Agent 上下文
```

- `GRILL.md` **不能**是 `profile.json` 里的一个字符串字段。理由:整条设计的前提是"人写的高信号 > LLM 推的",而人只会去改一份真 markdown;埋进 JSON 里满屏 `\n` 转义就等于永远没人改,降级链最上面那一档会一直是空的。
- `notes/` 单独落盘的价值在成本结构:分批精读是最贵的一段(约 150K token 输入),汇总只是重组。改汇总 Prompt 时重跑不该重付精读的钱。

**4. 生成流程分四段,其中第二段可短路**

- **S0 扫描**(纯代码):目录树、全量文件清单、约定文档收集、用户输入读取。
- **S1 证据分级**(纯代码):按 `用户输入 > 约定文档 > README > 清单文件 > 源码` 排序;约定文档信号充足时**跳过 S2**。
- **S2 分批精读**(LLM × N,可并行):每批产出结构化笔记。
- **S3 汇总**(LLM × 1):产出 `GRILL.md` + `profile.json`。
- 短路判据 MVP 先用简单阈值(存在 `CLAUDE.md`/`AGENTS.md` 且总长超过一个下限),后续按评估调。这是拍的,spec 只约束"存在短路机制",不写死阈值。

**5. 约定文档自被扫描目录向上遍历至文件系统根逐层收集**

- 每层试读约定文档,读不到即跳过(一次 ENOENT,无成本)。
- 类型不限于 `CLAUDE.md`:覆盖各家 AI 编码工具的规则文件(`CLAUDE.local.md`、`AGENTS.md`、`.cursorrules`、`.clinerules`、`.windsurfrules`,以及嵌套的 `.github/copilot-instructions.md` 与 `.cursor/rules/` 下规则)。参照 Claude Code `/init` 命令 survey 的规则文件清单——它们同样是"人手写的高信号约定",是证据分级第二档该抓的东西。原文与译文见下:
  > If there are Cursor rules (in .cursor/rules/ or .cursorrules) or Copilot rules (in .github/copilot-instructions.md), make sure to include the important parts.
  >
  > 译:如果存在 Cursor 规则(在 .cursor/rules/ 或 .cursorrules 中)或 Copilot 规则(在 .github/copilot-instructions.md 中),务必纳入其中的重要部分。
- 理由:约定文档常写在 monorepo 根上而非子仓内。扫描 `WhatIf/frontend` 时,`WhatIf/CLAUDE.md` 必须能被看见。
- 收集后**由根向下**排序注入,让更靠近被扫描目录的约定排在更后面。

**6. 分批的单位是目录,不是文件,也不是字节**

- 以目录为原子单位聚类,小目录合并至接近批预算,超预算的目录再按子目录拆;单个超大文件(如 49 KB 的 `runtime/game.py`)**单独成批且不截断**。
- 理由:S2 的判据是"只记跨文件才能得出的结论"。按文件切物理上产不出跨文件结论;按字节顺序切会把无关模块凑到一批,同样写不出东西。
- 现有的 `MAX_SOURCE_CHARS`(单文件截前 4000 字符)随之废除——截断一个核心文件产出的笔记是有毒的,宁可它独占一批。

**7. 地图只写"读多个文件才能得出"的结论**

- 判据固定为一句:**删掉这行,下游会不会误判?** 不会就砍。
- 排除:逐文件清单、构建/测试/lint 命令、代码风格、语言通用常识、材料里没有依据的内容。
- 保留:跨模块调用契约、架构决策与取舍、不显然的坑、代码与用户自述不一致处。
- 目录树**逐字附上**,不由 LLM 复述——它是"证明某物不存在"的依据,转述必然有损。小仓(如 WhatIf 双仓 3.4 KB)直接内联,代价可忽略;巨型项目(如 UE 超大仓,目录树可达 MB 级)则外置为 `GRILL.md` 引用的独立文件(`trees/<仓名>.txt`),正文只放引用。参照 Claude Code `/init` 用 `@path/to/import` 引用大内容而非内联的做法——阈值内内联(自包含),超阈值引用(正文精简),两种情形下逐字与不经 LLM 的性质都不变。

**8. 失败与盲区显式化**

- 批次失败记入 `failedBatches[]`,并在 `GRILL.md` 里写明该区域未被分析。
- 理由:精简的地图擅长指向存在的东西,不擅长证明不存在。若盲区不可见,下游会把"没写"读成"没有",这正好是最容易造成误判的一类错误。
- 与前置变更"单模块失败置占位继续"的差别:一个文件的职责丢了无关紧要,**一整批丢了等于一个模块从地图上消失**,不能同等对待。

**9. LLM 调用参数显式化**

- 现状是只发 `model` + `messages`,`temperature`/`max_tokens`/`response_format` 全走 provider 默认(温度 1.0、4096 上限、无结构约束)。
- S2 用低温 + JSON 结构约束;S3 需要更大的输出上限(地图正文 + metadata)。具体数值属实现细节,不入 spec。

**10. `links` 泛化为可选的 n 元关系**

- 从"必填的前端文件 ↔ 后端接口"改为可选的 `{ repos[], relation, source: user | inferred }`,并新增自由文本 `description`。
- 理由:单仓项目不该被迫编造二元链路;而结构化形态仍需保留(纯自由文本无法驱动模块挑选,下游也无法精确定位)。用户不填时由阅读者推导并标记 `source: inferred`。

**11. 超限时的响应方式按工具性质区分**

- `read_file` 超体积上限**抛错**,`grep`/`glob` 超条数上限**截断**。
- 依据是 Claude Code 的实测结论(`FileReadTool/limits.ts` 头部注释):

  > Tested truncating instead of throwing for explicit-limit reads that exceed the byte cap (#21841, Mar 2026). Reverted: tool error rate dropped but mean tokens rose — the throw path yields a ~100-byte error tool-result while truncation yields ~25K tokens of content at the cap.
  >
  > 译:针对超出字节上限的显式范围读取,试过用截断替代抛错(#21841,2026 年 3 月)。已回滚——工具报错率确实降了,但平均 token 消耗上升了:抛错这条路返回的是约 100 字节的错误结果,而截断在上限处返回的是约 25K token 的内容。

- 即抛错比截断省两个数量级的上下文,前提是错误信息**可执行**——所以 `read_file` 必须同时支持范围读取,否则"请改用范围读取"这句提示无处落地。
- 但该结论不能推广到搜索类工具:`grep`/`glob` 本就预期多命中,抛错等于什么都没返回。
- 截断信息只在**真截断**时给。恰好等于上限而未丢弃任何结果时报告截断,会诱导下游翻一页空结果。
- 解除上限的逃生口是"证明不存在"的前提:默认上限之下,被丢弃的结果里是否仍有命中无法排除,该判定不成立。这正是精简地图那条固有张力(擅长指向存在,不擅长证明不存在)在工具层的落点。

**12. 阅读者 agent 用 AI SDK 原生 tool-calling,不引 Mastra**

- 实现:`generateText` + `tools`(glob/grep/read_file)+ `stopWhen: stepCountIs(N)` 构成 agentic 循环,零新依赖(`ai@5` 已在依赖里)。
- 备选:proposal 原文拟引入 Mastra 做"角色与工具注册",与 `judge-agent-and-eval` 共用 —— 弃用。理由:
  - spec 是**框架无关**的("信息边界由工具注册与返回值类型决定"),没有任何一条要求 Mastra。
  - 阅读者是**无状态一问一答**(`ask_reader`),用不上 Mastra 的 workflow / memory / 多 agent 编排;为一个薄 agent 引一整套框架是过度设计。
  - 兼容性风险:Mastra 与 `ai` 版本历来有摩擦,tasks 5.1 本身就把"确认与 ai@5 兼容"列为待验风险;原生实现直接规避。
  - 共用收益暂时无法兑现:`judge-agent-and-eval` 尚未动工,且其设计里裁判**不注册任何文件工具**,未必需要 agent 框架。若确有需求,由该变更自行评估,不由本变更预先绑定依赖。

## Risks / Trade-offs

- **[阅读者把源码逐字转述进自然语言答案]** → 类型边界只能挡住"字段级"泄露,文本内容挡不住。Prompt 明确约束 + 评估期人工抽查若干条 `ask_reader` 回答。
- **[地图过于精简,裁判反复追问导致延迟与成本上升]** → 记录每场的 `ask_reader` 调用次数作为地图质量的反向指标;次数持续偏高说明地图该加密度,而不是该限流。
- **[短路判据过于粗糙,该精读的项目被跳过]** → MVP 用简单阈值,spec 不写死;S3 汇总时输出自评信心值,低于阈值可回头补 S2 留作后续增强。
- **[分批切断了跨批关联]** → 分批的固有代价。靠笔记里的"存疑"字段与 S3 汇总接线兜底;真接不上的,正是裁判该问阅读者的问题——在线问答本就是这个缺口的补丁。
- **[`grep` 返回结果过大撑爆上下文]** → 设结果条数上限并在截断时明确告知。上限之下无法排除"被丢弃的结果里仍有命中",故须留一个解除上限的显式入参供穷尽检索使用。
- **[产出契约变更导致 `judge-agent-and-eval` 返工]** → 该变更 0 行代码,现在改代价最低;本变更定稿后立即同步修订其工具集与输入契约。
- **[简历与 JD 在本变更只读取不使用]** → 终评在后续变更,本变更只负责读入与持有,避免后续再改一次输入契约。

## Migration Plan

- 无线上系统,无数据迁移。既有 `light-profile.json` 产物直接作废重新生成。
- CLI 参数从 `--frontend/--backend/--links/--out<文件>` 迁移到以仓库列表 + `--out<目录>` 为中心;`--links` 变为可选。
- 落地顺序:先改产出形态与生成流程(离线),再加 `ask_reader`(在线),最后同步修订 `judge-agent-and-eval`。

## Open Questions

- `GRILL.md` 的体积上限:按 100K 上下文暂定 12000 字符量级,需按真实地图与裁判表现校准。
- S2 的批预算:实测双仓可切 5 批,但批越大跨文件结论质量越高、越接近上下文上限,需要试。
- `ask_reader` 是否需要限次:MVP 不限,先观察实际调用次数再定。
- 阅读者推导 `links` 的准确率是否可接受,还是仍以用户手写为主。
