## Context

面试模块 MVP 需要一份小到能整个塞进裁判上下文的项目速览(轻档案)。当前状态是零基础:没有任何扫描脚本,也没有轻档案格式约定。本变更是整个 MVP 的第一步,纯数据管道,不引入 Mastra 编排或任何 Agent 角色。约束:

- 技术栈固定为 TypeScript + Zod;LLM 调用走 Vercel AI SDK,与后续 Agent 模块保持一致。
- LLM provider 定为 **DeepSeek**(`@ai-sdk/deepseek`,模型 `deepseek-chat`),环境变量 `DEEPSEEK_API_KEY`。理由:模块职责生成是高频、低难度的调用,DeepSeek 国内直连且成本低;走 Vercel AI SDK 接入,后续要换厂商只需替换 provider 一行。
- 轻档案是后续「裁判 Agent」变更的直接输入,其 JSON 字段结构一旦定下需保持稳定。
- 被扫描的前后端仓是本地路径,离线可读。

## Goals / Non-Goals

**Goals:**

- 一条命令从"前端仓 + 后端仓 + 链路配置"生成一份通过 Zod 校验的轻档案 JSON。
- 目录树精简、可控体积,确保能整体放进裁判上下文。
- 模块职责由 LLM 生成,`links` 由用户手写并做存在性校验。
- 脚本可独立运行、独立验证,不依赖任何后续模块。

**Non-Goals:**

- 不做自主下钻、预算护栏、跨仓库链路自动识别(完整版的事)。
- 不引入向量库(Qdrant)——MVP 阶段轻档案不大,无需检索。
- 不做增量/缓存更新;每次全量重新生成即可。
- 不含任何 Agent、编排或面试流程逻辑。

## Decisions

**1. 轻档案结构与生成方式的分工**
- `project_name`、`repos[].name/path/tree`:脚本机械生成。
- `repos[].modules[].role`:LLM 生成一句话职责。
- `links`:用户手写配置,脚本只校验+合并。
- 理由:机械能算的绝不花 LLM 成本;只有"职责概括"这类需要理解语义的才用 LLM。

**2. 用 Zod 作为唯一结构真源**
- 定义 `LightProfileSchema`,生成后 `parse` 自校验,通过才落盘。
- 备选:手写类型 + 运行时断言 —— 弃用,Zod 一处定义即得类型 + 运行时校验,且与后续裁判的 `JudgeOutput` schema 风格统一。

**3. 目录树扫描的忽略策略**
- 维护一份忽略清单(`node_modules`、`dist`、`build`、`.git`、`.next`、`coverage` 等),扫描时跳过。
- 备选:读取仓库 `.gitignore` —— 后续可增强,MVP 先用固定清单,简单可控。

**4. 模块识别的粒度**
- MVP 由"关键模块"驱动:优先对 `links` 中出现的文件、以及少量入口文件生成 `role`,而非对全部源码文件逐个调用 LLM(成本与体积双爆炸)。
- 理由:轻档案的价值是"给裁判找接缝的弹药",链路相关文件信息密度最高。

**5. LLM 调用的容错**
- 单个模块 `role` 生成失败降级为占位并告警,不中断整体流程。
- 理由:预处理是一次性离线任务,局部失败不该让整场重跑。

## Risks / Trade-offs

- **[LLM 生成的 role 不准/含幻觉]** → 轻档案仅作"出题弹药"提示,裁判仍以 `read_file` 读真实源码为准;role 不准最多降低出题精度,不影响判断正确性。
- **[固定忽略清单漏掉某仓库的特殊产物目录]** → 轻档案可能偏大;先观察实际体积,必要时再补清单或接 `.gitignore`。
- **[关键模块识别过窄,漏掉重要文件]** → 裁判可通过 `read_file` 按目录树自行下钻补足;role 缺失不阻断。
- **[LLM 调用成本]** → 通过只对关键模块生成 role 控制调用次数;预处理是一次性任务,成本可接受。
