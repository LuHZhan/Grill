## Why

面试模块 MVP 的裁判 Agent 需要"真实代码"作为出题弹药,但直接把整个前端仓 + 后端仓塞进上下文既不现实也太贵。我们需要一份**预先生成、结构化、体积小到能整个放进裁判上下文**的项目速览——即"轻档案"。它是整个面试模块的数据地基:没有它,裁判无从下钻,面试官也缺少项目背景。这是 MVP 的第一个开发里程碑,纯数据管道,不含任何 Agent,可独立跑通与验证。

## What Changes

- 新增一个**独立预处理脚本**(命令行运行),输入前端仓路径 + 后端仓路径 + 用户手写的功能链路配置,输出一份 JSON 轻档案。
- 轻档案结构包含:`project_name`、`repos[]`(每个含 `name`/`path`/精简 `tree`/`modules[]`)、`links[]`(前后端功能链路)。
- 每个 `module` 的一句话职责由 **LLM 扫描源码生成**;`links` 由用户手写配置提供,脚本只做校验与合并。
- 目录树生成时**忽略 `node_modules`、构建产物、`.git` 等噪声**,保证轻档案足够精简。
- 用 **Zod schema** 约束轻档案输出结构,防止字段漂移;生成后自校验通过才落盘。
- 轻档案落盘为单个 JSON 文件,供后续裁判 Agent 变更直接读取。

## Capabilities

### New Capabilities

- `light-profile`: 扫描前后端仓库并生成结构化 JSON 轻档案的能力,含目录树精简、模块职责的 LLM 生成、用户链路配置合并,以及 Zod schema 校验与落盘。

## Impact

- **新增代码**:预处理脚本及其 Zod schema 定义、目录树扫描工具、模块职责的 LLM 生成模块、链路配置读取与校验。
- **新增配置**:用户手写的功能链路配置文件(前端文件/函数 ↔ 后端接口的对应关系)。
- **技术栈**:TypeScript + Zod;LLM 调用走 Vercel AI SDK(与后续 Agent 模块一致)。
- **产出契约**:轻档案 JSON 的字段结构,是后续「裁判 Agent + 评估」变更的直接输入,须保持稳定。
- **无 Agent、无编排依赖**:本变更不引入 Mastra 编排,可单独交付。
