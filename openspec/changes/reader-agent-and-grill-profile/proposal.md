## Why

面试模块的三个角色分工现已确定:**面试官**只知道简历与岗位 JD,**裁判**只知道项目地图,**阅读者**全知(源码 + JD + 简历)。"全知"这件事此前无处安放——它被错误地假设在裁判身上,而裁判一旦能读源码,"面试官物理隔离于源码"这条边界就只剩一层。

现有 `light-profile` 是一条纯脚本管道,不是角色。它的产出在真实仓上已暴露两个硬伤:

- **逐文件生成的 `role` 没有信息量**。对 WhatIf 双仓实测 11 条职责,11 条都是"读单个文件就能得到"的复述(如"渲染 React 主应用并挂载到 DOM 根节点"),对下游等同于噪声。根因是形式本身——单文件视角 + 只读前 4000 字符 + 30 字上限,物理上产不出跨文件结论。
- **关键模块识别偏向入口文件**。backend 105 个文件只挑出 5 个,全是路由与入口;真正承载技术含量的 `runtime/game.py`(49 KB,全仓最大)与 `preprocessing/entity_transition/` 完全不在档案里。地图上只标了大门和走廊,没标房间。

同时,裁判被限定为只读地图,却必然遇到地图答不了的问题("他说做了断线重连,到底有没有")。这类问题需要全知者现场作答,而不是把地图撑成百科全书。

阅读者还是后续**项目学习**链路的核心角色,其能力不只服务于面试。

## What Changes

- 新增**阅读者 Agent**:全知角色,注册 `glob` / `grep` / `read_file` 三个工具,输入包含被测项目源码、用户提供的简历履历文件与岗位 JD 文件。
- **离线职责**:产出改为一个目录而非单个 JSON —— `GRILL.md`(项目地图,markdown 正文,裁判整篇读)+ `profile.json`(metadata)+ `notes/`(分批笔记,仅用于调试与增量复跑,不进任何 Agent 上下文)。
- **在线职责**:提供 `ask_reader(question)` 接口,面试中由裁判调用。返回值 **MUST 是自然语言结论,不得含源码片段或代码块**——该边界由返回值类型保证,不靠 Prompt 约定。
- **生成方式改为分阶段 map-reduce**:扫描 → 证据分级 → 分批精读 → 汇总。证据分级遵循"人写的优先"降级链:用户输入 > 约定文档(`CLAUDE.md`/`AGENTS.md`/`README` 等,自被扫描目录向上遍历至根逐层收集) > 清单文件 > 源码;约定文档信号充足时可跳过分批精读。
- **地图的取舍判据写死为一条**:每一行都要通过"删掉这行,下游会不会误判"。只写需要读多个文件才能得出的结论;逐文件清单、构建/测试命令、代码风格一律排除——前者下游自己检索即可,后者与判断无关。
- **目录树原样附在 `GRILL.md` 末尾,不由 LLM 复述**。它承担"证明某个东西不存在"的职责,转述必然有损。实测双仓目录树合计 3.4 KB,代价可忽略。
- **BREAKING** —— `LightProfileSchema` 结构变更:废除 `repos[].modules[].role` 的逐文件生成,产出契约从"单个 JSON 文件路径"改为"输出目录"。已实现的逐文件职责生成与 `MAX_SOURCE_CHARS` 截断一并移除。
- **BREAKING** —— `links` 不再是必填的前后端二元关系:改为可选,支持 n 元(泛化 `repos[]` + `relation`),可由用户提供或由阅读者推导;并新增自由文本 `description` 字段。单仓项目不再需要手写链路配置。
- **新增 `contradictions[]`**:预处理阶段发现的"用户自述与代码证据不一致"原样记录,不做修正——它是下游最有价值的输入。
- **失败不得静默**:分批精读失败的批次记入 `failedBatches[]`,并在 `GRILL.md` 中显式标注该区域未被分析。地图上的空白必须可见,否则会被误读为"那里不存在东西"。
- LLM 调用参数显式化(`temperature` / `max_tokens` / `response_format`),不再依赖 provider 默认值。

**不在本变更范围内**:阅读者的第三项职责「终评」(综合两份评价 + 代码 + JD + 简历)需要面试官评价存在,留到后续变更;面试官角色、编排层、复盘同理。

## Capabilities

### New Capabilities

- `reader-agent`: 阅读者角色本身——全知的信息边界、`glob`/`grep`/`read_file` 工具集与路径安全约束、`ask_reader` 在线问答接口及其"只回自然语言结论"的返回值边界。

### Modified Capabilities

- `light-profile`: 产出形态由单个 JSON 改为 `GRILL.md` + `profile.json` + `notes/` 目录;生成方式由逐文件 LLM 职责改为分阶段 map-reduce 与证据分级;`links` 由必填二元关系改为可选 n 元关系;新增 `contradictions[]` 与 `failedBatches[]`;新增简历与 JD 两份输入文件。

## Impact

- **改造代码**:`src/preprocess/` 现有模块。目录树扫描、忽略清单、路径校验、Zod schema 与 CLI 骨架可复用;`modules.ts` 的逐文件职责生成整体废弃。
- **新增代码**:阅读者 Agent 定义、`glob`/`grep`/`read_file` 三工具及其仓根内路径校验、`ask_reader` 接口、约定文档的向上遍历收集、分批切分与汇总、`GRILL.md` 渲染。
- **新增输入**:用户提供的简历履历文件与岗位 JD 文件。
- **新增依赖**:Mastra(角色与工具注册)——与 `judge-agent-and-eval` 变更共用,先落地者引入。
- **产出契约变更**:下游 `judge-agent-and-eval` 消费的不再是 `LightProfile` JSON,而是 `GRILL.md` + `profile.json`;裁判的 `read_file` 工具须替换为 `ask_reader`。该变更需在本变更定稿后同步修订。
- **成本**:分批精读对 WhatIf 双仓约 150K token 输入、5 批调用,`notes/` 缓存使重跑汇总阶段不必重付精读成本。
