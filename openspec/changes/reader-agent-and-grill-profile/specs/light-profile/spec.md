## MODIFIED Requirements

### Requirement: 轻档案生成命令

系统 SHALL 提供一个命令行入口,接收一个或多个被测仓库路径、用户简历履历文件路径、岗位 JD 文件路径,以及一个**可选**的功能关系配置文件路径,在指定的**输出目录**下生成项目地图与 metadata。

输出目录 MUST 包含:

- `GRILL.md` —— 项目地图正文,markdown,供下游整篇读入
- `profile.json` —— 结构化 metadata
- `notes/` —— 分批精读产生的中间笔记

#### Scenario: 提供合法输入时生成产物目录

- **WHEN** 用户以合法的仓库路径、简历履历文件与岗位 JD 文件运行阅读者的离线生成
- **THEN** 系统在输出目录下写出 `GRILL.md`、`profile.json` 与 `notes/`,并打印各产物路径与体积

#### Scenario: 未提供关系配置时仍能生成

- **WHEN** 用户未提供功能关系配置文件
- **THEN** 系统正常生成产物,关系由阅读者推导或留空,不因缺少该配置而失败

#### Scenario: 仓库路径不存在时报错退出

- **WHEN** 传入的任一被测仓库路径在文件系统中不存在
- **THEN** 系统以非零退出码终止,并打印指明哪个路径无效的清晰错误信息,不生成任何输出文件

### Requirement: 精简目录树扫描

系统 SHALL 为每个仓库生成精简的目录树,MUST 忽略依赖目录、构建产物目录、版本控制目录及其他噪声。忽略清单 MUST 覆盖:

- 后缀不足以识别的构建产物目录(如 Electron 的 `dist-electron`)
- 后缀伪装成源码的锁文件(如 `package-lock.json`、`pnpm-lock.yaml`、`bun.lockb`、`go.sum`)
- 样式文件——用了什么样式方案可从依赖清单看出,样式源码本身对项目理解零信息量

后缀匹配 MUST 大小写不敏感。

生成的目录树 MUST 原样附在 `GRILL.md` 末尾,MUST NOT 由 LLM 复述或改写——目录树是判断"某个东西是否存在"的依据,转述会造成有损。

#### Scenario: 扫描时忽略噪声目录

- **WHEN** 仓库中包含 `node_modules`、`dist`、`dist-electron` 或 `.git` 目录
- **THEN** 生成的目录树中不包含这些目录及其内容

#### Scenario: 忽略后缀伪装成源码的锁文件

- **WHEN** 仓库根目录下存在 `package-lock.json` 或 `pnpm-lock.yaml`
- **THEN** 这些文件既不出现在目录树中,也不进入待精读的文件清单

#### Scenario: 目录树原样进入地图

- **WHEN** 系统生成 `GRILL.md`
- **THEN** 扫描得到的目录树以原文形式出现在 `GRILL.md` 中,内容与扫描结果逐字一致

### Requirement: 用户功能关系配置合并与校验

系统 SHALL 支持一个**可选**的功能关系配置。每条关系 MUST 支持引用任意数量的仓库内路径(不限于前端与后端两方),并 MUST 标记其来源是用户提供还是阅读者推导。

系统 SHALL 校验每条关系引用的路径存在于对应仓库,缺失时告警并跳过该条,不中断整体流程。

用户未提供配置时,阅读者 MAY 自行推导关系,推导所得 MUST 标记为推导来源。

#### Scenario: 合法关系配置合并成功

- **WHEN** 用户提供的关系配置中每条关系引用的路径都能在对应仓库中找到
- **THEN** 系统将这些关系合并进 metadata,并标记来源为用户提供

#### Scenario: 关系引用不存在的路径时报警

- **WHEN** 某条关系引用的路径在仓库中不存在
- **THEN** 系统打印指明该条关系与缺失路径的告警信息并跳过该条,继续处理其余关系

#### Scenario: 单仓项目无需关系配置

- **WHEN** 被测项目只有一个仓库,用户未提供关系配置
- **THEN** 系统正常生成产物,不要求用户编造关系

### Requirement: 产出结构由 Zod schema 约束

系统 SHALL 使用 Zod schema 定义并校验 `profile.json` 的结构。该结构 MUST 包含 `project_name`、`repos`、`links`、`entrypoints`、`open_questions`、`contradictions` 与 `failed_batches` 字段;只有通过 schema 校验的 metadata 才允许落盘。

`GRILL.md` 的正文 MUST 作为独立的 markdown 文件落盘,MUST NOT 作为 `profile.json` 中的字符串字段存储——地图需要被用户直接阅读与修改,嵌入 JSON 会使其实际上不可编辑。

#### Scenario: 结构合法时通过校验并落盘

- **WHEN** 生成的 metadata 对象满足 Zod schema 定义的全部字段与类型约束
- **THEN** 系统通过校验,将 metadata 序列化写入 `profile.json`,并将地图正文写入独立的 `GRILL.md`

#### Scenario: 结构不合法时拒绝落盘

- **WHEN** 生成的 metadata 对象缺少必填字段或字段类型不符
- **THEN** 系统以非零退出码终止并打印 schema 校验错误,不写出损坏的输出文件

## ADDED Requirements

### Requirement: 证据分级与约定文档收集

系统 SHALL 在生成地图前收集项目中人工撰写的约定文档,并按可信度分级使用:用户输入 > 约定文档 > README > 清单文件 > 源码。

约定文档的收集 MUST 自每个被扫描目录向上逐层遍历至文件系统根,每层尝试读取约定文档(如 `CLAUDE.md`、`AGENTS.md`、`README.md`),读取不到即跳过。收集结果 MUST 按由根向下的顺序排列。

当约定文档提供的信号已足够时,系统 MAY 跳过分批精读阶段。

#### Scenario: 收集到 monorepo 根目录的约定文档

- **WHEN** 被扫描目录是 monorepo 的子目录,而约定文档位于该 monorepo 的根目录
- **THEN** 向上遍历过程中读取到该约定文档,并将其纳入证据

#### Scenario: 无约定文档时降级到源码

- **WHEN** 自被扫描目录向上至根均未找到任何约定文档
- **THEN** 系统进入分批精读阶段,以源码作为主要证据

### Requirement: 分批精读产出跨文件结论

当需要精读源码时,系统 SHALL 将待读文件按目录聚类分批,每批一次 LLM 调用,产出该批的结构化笔记。

分批 MUST 以目录为原子单位,MUST NOT 按单个文件或按字节顺序切分——单文件视角物理上产不出跨文件结论,按字节切分会把互不相关的模块凑进同一批。

单个文件体积超过批预算时 MUST 单独成批,MUST NOT 截断其内容。

每批笔记 MUST 落盘到 `notes/`,以便重跑汇总阶段时复用,不必重付精读成本。

#### Scenario: 按目录聚类分批

- **WHEN** 系统准备精读一个含多个模块目录的仓库
- **THEN** 同一目录下的文件被分入同一批,批的规模受批预算约束

#### Scenario: 超大单文件单独成批且不截断

- **WHEN** 某个源码文件的体积超过单批预算
- **THEN** 该文件单独成为一批,其内容被完整读入,不被截断

#### Scenario: 笔记落盘供重跑复用

- **WHEN** 分批精读完成
- **THEN** 每批笔记以独立文件写入 `notes/`,重新执行汇总阶段时可直接读取,不重新调用 LLM 精读

### Requirement: 地图只记录跨文件才能得出的结论

`GRILL.md` SHALL 是一份指路的地图而非项目百科。其内容的取舍 MUST 遵循一条判据:**删掉这一行,下游会不会因此误判**;不会则 MUST NOT 写入。

地图 MUST NOT 包含逐文件的清单或组件表、构建/测试/lint 命令、代码风格约定,以及所读材料中没有依据的内容。

地图 SHALL 包含跨模块的调用契约、架构决策与取舍、不显然的坑,以及用户自述与代码证据不一致之处。

#### Scenario: 排除单文件即可得到的信息

- **WHEN** 某项信息读取单个文件即可获知(如某文件定义了哪个类型)
- **THEN** 该信息不写入 `GRILL.md`

#### Scenario: 保留跨文件才能得出的结论

- **WHEN** 某项结论需要读取多个文件才能得出(如两个模块对同一状态的处理阈值不一致)
- **THEN** 该结论写入 `GRILL.md`

### Requirement: 记录用户自述与代码证据的不一致

系统 SHALL 将生成过程中发现的"用户自述与代码证据不一致"原样记录进 metadata 的 `contradictions` 字段,每条 MUST 同时保留用户的说法与对应的代码证据。

系统 MUST NOT 依据代码自行修正用户的说法后丢弃该不一致——不一致本身是下游最有价值的输入。

#### Scenario: 用户自述与依赖清单矛盾时记录

- **WHEN** 用户自述使用了某项技术,而项目的依赖清单与源码中均无对应痕迹
- **THEN** 系统在 `contradictions` 中记录该说法与对应证据,不修改用户的原始说法

### Requirement: 分析盲区必须显式可见

当某一批精读失败时,系统 SHALL 将该批记入 metadata 的 `failed_batches` 字段,并在 `GRILL.md` 中显式标注该区域未被分析。

系统 MUST NOT 静默跳过失败的批次——地图上不可见的空白会被下游读作"那里不存在东西",而这正是最容易造成误判的一类错误。

#### Scenario: 批次失败时在地图中标注盲区

- **WHEN** 某一批的精读调用在重试后仍然失败
- **THEN** 该批记入 `failed_batches`,且 `GRILL.md` 中出现一处明确说明该区域未被分析的标注

#### Scenario: 全部批次失败时拒绝落盘

- **WHEN** 所有批次的精读调用都失败
- **THEN** 系统以非零退出码终止并打印各批的失败原因,不写出一份没有任何分析结果的地图

## REMOVED Requirements

### Requirement: 模块职责由 LLM 生成

**Reason**: 逐文件生成一句话职责的形式产不出有价值的信息。对真实仓实测的 11 条职责全部是"读单个文件即可得到"的复述,对下游等同于噪声;根因是形式本身——单文件视角、只读文件开头固定长度、输出长度上限三者叠加,物理上无法产出跨文件结论。同时"关键模块"的入口文件启发式使全仓最大、技术含量最高的文件落选。

**Migration**: 由「分批精读产出跨文件结论」与「地图只记录跨文件才能得出的结论」两条需求取代。逐文件的 `role` 字段与单文件源码截断上限一并移除;下游需要单个文件的信息时,由阅读者的 `read_file` 工具或 `ask_reader` 接口按需提供。
